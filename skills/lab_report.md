# Lab Report Skill SOP

本 SOP 指导 Coder agent 在完成实验报告类作业时的实验复现 / 数据组织 / 报告撰写。
ground truth 全部从 task context（intake_result + 实验指导原文 + 用户提供的实验环境）拿；
绝不允许编造未跑过的实验数据 / 未截过的截图 / 未发生的实验现象。

## 1. 读懂实验指导（先做这一步）

调 sandbox_convert_to_markdown 把 PDF / DOCX 实验指导转 markdown 后读，抽：
- 实验目的（一句话总结要验证什么 / 实现什么）
- 实验环境（OS / 容器 / 网络拓扑 / 工具版本）
- 必做任务清单（Task 1.1, 1.2, ... 通常分主任务 + 子任务）
- 加分任务（Bonus / Optional Task）
- 提交要求（章节齐？复现说明？截图证据？）
- 评分细则（章节占比、扣分项）

本 skill 期望产物**章节齐 + 每个 Task 都有"实验过程 + 现象 + 结论"完整链条**，缺一不可。

## 2. 实验环境配置（在主报告之前先 doc 它）

主报告第一节讲清楚环境，后面才能复现。重点：
- 容器 / VM 拓扑（多容器要画出 IP / 网络示意）
- docker-compose.yml 里的关键改动（如 network_mode: host 这类常见 WSL 适配）
- 容器需要的特殊权限（privileged / cap_add / seccomp 等）

如果实验需要改 docker / 系统配置才能跑通，**单独写一段「环境前置」描述**，让助教知道"为什么我改了 yml"。
样例：

> 在一切开始之前，因为 docker 容器配置问题，需要修改 docker-compose.yml 让容器能联网下载部件：
> ```yaml
> services:
>   attacker:
>     build:
>       context: ./image_ubuntu_mitnick
>       network: host
> ```
> 网络改成主机网络，宿主机开 VPN 即可正常访问外网。
> 另外还需要开启容器的 privileged 权限。

## 3. 实验报告必备章节（Verifier 阶段 1 硬指标）

```
1. 实验目的
2. 实验原理（公式 / 关键概念 / 协议规范引用）
3. 实验环境（OS / 容器 / 网络拓扑 / 工具 + 版本）
4. 实验步骤（按 Task 1.1 / 1.2 / ... 分小节，每节含命令 + 现象 + 截图占位 + 思考）
5. 实验结果（数据表 + 图，含原始记录）
6. 结果分析与讨论（含为什么这样、与预期对比、有无意外现象）
7. 结论
8. 思考题（如实验指导附了思考题）
```

Verifier 会扫这 5 个关键字至少齐：实验目的 / 实验原理 / 实验步骤 / 实验结果 / 结论。

## 4. 实验过程章节的写作风格（这是产物的灵魂）

**这是 lab_report 与 coding / essay 最大的差异**：lab_report 必须**展现思考过程**，
不是堆命令 + 截图。每一步前要 1-2 句解释「我为什么这么做、考虑过什么 alternative」。

格式参考：

```
首先启动容器，进入受害者容器 10.9.0.5 关闭 SYN Cookie 防护并清除 TCP 缓存。

这里我同时减小了半开连接队列的大小，使得攻击更容易成功。

（此处建议附受害者容器关 SYN Cookie + 改队列大小 + 清缓存命令的截图）

在这个受害者容器中同时开启监控半连接队列：

（此处建议附 ss -tnl / netstat -an 监控输出的截图）

接下来开始攻击，synflood.py 已经完成，进入攻击者容器执行攻击脚本即可：

（此处建议附攻击执行截图）

受害者容器中监控到半连接：

（此处建议附半连接队列被填满的截图）

在可信服务器中尝试 telnet 到受害者失败，说明攻击成功。

（此处建议附 telnet 超时截图）
```

注意要点：
- 每一步先用 1-2 句话解释**思考与抉择**（"我同时减小了队列大小，使攻击更容易成功"）
- 截图位置用 `（此处建议附 XX 截图）` 占位，不直接插图（agent 不能产截图，由用户补）
- **少用加粗**（用户偏好；加粗只用于真正强调点，如关键发现 / 结论判断）
- 代码块附在文中（不要单独成节），命令前后都要有上下文说明

## 5. 代码 / 攻击脚本附在文中

如果实验含代码（如 synflood.py / tcp_rst.py），**整段贴在对应步骤文中**，不要附录式塞最后。
代码块前一句说明「这段代码做什么 + 关键参数选择理由」：

```python
# tcp_rst.py
#!/usr/bin/env python3
from scapy.all import *

ip = IP(src="10.9.0.5", dst="10.9.0.6")
tcp = TCP(sport=23, dport=33924, flags="R", seq=1987657081)
pkt = ip/tcp
send(pkt, verbose=0)
print("RST packet sent!")
```

伪造一个从 10.9.0.5 发往 10.9.0.6 的 RST 包，seq 设置为 10.9.0.6 期望的值（即 ACK 值）。

## 6. 实验结果与分析

**结果**：原始记录（命令输出 / 数据表 / 实测耗时）。能上 markdown table 的就上：

| Task | 是否成功 | 耗时 | 关键现象 |
|---|---|---|---|
| 1.1 (py SYN flood) | ✓ | ~30s | 队列填满，telnet 超时 |
| 1.2 (c SYN flood) | ✓ | ~5s | 比 py 快 6×，无 py 解释器开销 |
| 1.3 (开 SYN Cookie) | ✗（攻击失败合预期） | - | Cookie 不存半开状态，队列不会填满 |

**分析**：
- 与预期对比（理论应该 X，实测 Y，差异原因）
- 意外现象（如 py 比 c 慢 6× 这种性能观察，要解释为什么）
- 防御机制有效性证明（如 Task 1.3 攻击失败本身就是 SYN Cookie 有效的证据）

## 7. 引用规范（写作部分参考 essay skill）

实验指导原文可直接引用作分析对象（hwHandler 全局学术诚信约束第 2 条）。
其他来源（教材 / RFC / 论文）：
- 用引号 + 注明来源
- 单条直接引用 ≤ 30 字
- 必要时附 RFC 编号 / 论文 DOI

## 8. 学术诚信（与 hwHandler 全局约束一致）

- 同组同学讨论思路可以，代码 / 报告不能互抄
- 不展示他人学号 / 姓名 / 真实 IP / 真实凭据
- 用了 AI 辅助：末尾「使用工具说明」段标注（哪几节 AI 协助生成）

## 9. 自检 checklist（写完后逐条对照）

- [ ] 5 个关键章节齐（实验目的 / 实验原理 / 实验步骤 / 实验结果 / 结论）
- [ ] 实验环境章节有 OS / 容器 / 工具版本
- [ ] 每个 Task 都有"过程 + 现象 + 结论"完整链条
- [ ] 每一步前有 1-2 句思考解释（不是流水账）
- [ ] 截图占位 hook 在所有需要视觉证据的地方都打了（用 `（此处建议附 XX 截图）`）
- [ ] 代码块整段贴在文中而非附录
- [ ] 数据表 / 实测数字真实（不允许编造）
- [ ] 结果分析包含与预期对比 + 意外现象解释
- [ ] 末尾有「使用工具说明」段

## 10. 与 coding / essay skill 的边界（各司其职）

- 题面要求"实现 + 测试，无实验过程描述要求" → coding skill（产物是代码不是报告）
- 题面要求"做实验 + 写报告" → 本 skill（产物是 lab_report.md，含实验全过程）
- 题面要求"分析 / 论述 / 读后感"（无实验过程） → essay skill
- 实验报告中**可以**含 coding 类的代码 / 测试，但报告本身才是主交付物，不是单纯的代码包

## 异常处理速查

| 失败模式 | 应对 |
|---|---|
| 实验未实际跑通（容器起不来 / 命令出错） | **不要伪造结果**；在对应 Task 节写「实测未跑通：错误现象 + 已尝试方法 + 卡点」，让 Verifier fail，让用户接手 |
| 截图无法生成（agent 没有截图能力） | 全部用 `（此处建议附 XX 截图）` 占位，文末「## 使用工具说明」节明确说"截图待用户在实际运行后补充" |
| 实验数据有意外（如比预期慢 100×） | 这是 lab_report 的精华：在「分析」章节单独讨论意外现象 + 假设原因，比平铺数据更得分 |
| 实验指导是英文 SEED Lab 风格 | 报告主体仍用中文（用户偏好）；引用 SEED 原文时英文原文 + 中文意译 |
| 必备章节遗漏 | Verifier 阶段 1 会标 fail；自检 checklist 走一遍 |
